# Visible Output Contract

Use only user-visible facts and explicit assumptions. Do not expose private reasoning.

## brief

Use one or two sentences stating what will be done and the deliverable or boundary, then execute. Do not add an unrelated plan, question, or full grill.

## detailed-execute

State, concisely:

1. selected mode and the material reason;
2. goal and deliverable;
3. scope and non-goals;
4. explicit assumptions, key ambiguity, or risk;
5. execution and verification approach.

Then execute.

## detailed-clarify

Use the detailed structure, then ask exactly one question with one decision variable and state the outcome it changes. Do not combine independent unknowns into a checklist (for example, recipient, channel, and version). Choose the highest-value unknown first; offer up to three comparable options and a recommendation only when useful. Do not execute the answer-sensitive action first.

## safety-gated

State the material safety/approval boundary and the next safe action. Do not perform the restricted action without the required approval or clarification. If several facts are unknown, ask only the single fact whose answer most changes the safety decision; do not request a multi-field checklist.

## Silent ordinary execution

When the user explicitly asks to execute without visible understanding, omit every user-visible ordinary routing preface, including commentary that says the task was classified or routed. Return only the requested ordinary deliverable. Never omit an essential clarification, safety/approval boundary, or an explicit full grill.
